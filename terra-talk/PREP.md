## 3. history of terraform

this section exists to give people context for why terraform is the way it is. three sub-slides, keep the whole section under 3 minutes.

### hashicorp's founding (2012)

mitchell hashimoto and armon dadgar founded hashicorp in 2012. mitchell was already known for vagrant, which codified development environments. the founding thesis was that this same approach — defining infrastructure in code — could apply across the entire stack. hashicorp built consul (service discovery), vault (secrets management), nomad (workload scheduling), packer (image building), and terraform (provisioning). all of these share a common philosophy: declarative configuration, separation of intent from execution.

the key thing to internalize here: terraform didn't come out of nowhere. it came from a company whose entire identity was "infrastructure should be code." that philosophy is baked into every design decision.

### terraform 0.1 (july 2014)

terraform launched as version 0.1 in july 2014 under the mozilla public license 2.0 (a permissive open-source license). the key differentiator from day one was provider-agnosticism: at the time, aws cloudformation was the dominant iac tool, but it only worked with aws. terraform's bet was that teams would want a single workflow regardless of which cloud providers they used.

two design decisions were there from the start:
- **plan-then-apply**: you always preview what will change before it executes. this is still terraform's most important property.
- **hcl**: hashicorp configuration language, introduced as an alternative to writing raw json. it was designed to be human-readable but machine-parseable. yaml was not chosen because of its issues with implicit type coercion and whitespace sensitivity.

### the bsl relicense (august 2023)

in august 2023, hashicorp changed terraform's license from mpl 2.0 to the business source license (bsl) 1.1. this is worth understanding because people will ask about it.

what the bsl does: it prohibits using terraform to build a competing commercial product. companies like spacelift or env0 (terraform-as-a-service platforms) were affected. standard production use by end-user companies (like us) is explicitly permitted.

what happened next: within a month, the linux foundation announced opentofu, a community fork that maintains the mpl 2.0 license. opentofu aims for drop-in compatibility with terraform.

our position: the license change has zero practical impact on tabs. we're end users, not building a competing terraform service. we stayed on hashicorp terraform. if someone asks about opentofu, the honest answer is: it's a valid option and a good insurance policy, but the ecosystem (providers, modules, documentation) is still overwhelmingly oriented around terraform proper.

don't editorialize on whether the license change was good or bad. just cover the facts.

## 4. what terraform does

this section establishes shared understanding. the audience has some exposure to terraform, so you're not convincing anyone to use it — you're making sure everyone has the same mental model before getting into the details.

### infrastructure as code

four properties to hit:
- **declarative**: you describe the end state, not the steps. terraform figures out the ordering and api calls.
- **idempotent**: running the same config twice produces the same result. this makes it safe to re-run.
- **version-controlled**: because everything is in `.tf` files checked into git, infrastructure changes go through pull requests and code review, just like application code.
- **self-documenting**: the config files themselves are living documentation of what's deployed.

the shift from "clicking in the console" to "config files in git" is the fundamental value proposition. everything else — reproducibility, auditability, collaboration — flows from that.

### the workflow: init, plan, apply

three commands:
- `terraform init`: downloads provider plugins and configures the backend (where state is stored).
- `terraform plan`: reads your config, compares it to the current state, and shows a diff of what would change. nothing happens yet.
- `terraform apply`: executes the plan.

additional context worth knowing:
- there are over 3,000 providers covering cloud platforms, saas tools, dns services, monitoring, and more.
- hcl is a purpose-built configuration language — more readable than json, less ambiguous than yaml.
- the preview-before-execute model means you always know what will happen before it happens. this is one of the strongest arguments for terraform over imperative scripting approaches.

## 5. the building blocks (core concepts)

this is a vocabulary refresher. six concepts, roughly one minute each. if the audience seems comfortable, move through it quickly — the terragrunt section is where the new information lives for most people.

### providers

providers are plugins — separate go binaries — that terraform downloads during `init`. each one teaches terraform how to interact with a specific api.

- the aws provider is the most widely used and covers hundreds of resource types (ec2, s3, iam, cloudwatch, rds, etc.)
- we use aws for infrastructure, datadog for monitoring, and github for repository settings
- you can mix providers in the same config — a single `terraform apply` can provision an aws resource, set up its monitoring in datadog, and configure the github repo
- providers are version-pinned in the `required_providers` block to prevent breaking changes

### resources

resources are the core building block. each `resource` declaration maps to exactly one real infrastructure object.

the format: `resource "TYPE" "NAME" { ... }`
- the type prefix tells you the provider (`aws_`, `google_`, `azurerm_`)
- terraform tracks each resource in state by its address, like `aws_s3_bucket.logs`
- **important gotcha**: renaming a resource in config is interpreted as "destroy the old one and create a new one," not a rename. you can use `moved` blocks to handle this without destruction, but it catches people off guard.

### variables, outputs, locals

think of a terraform module like a function:
- **variables** = parameters. set via `-var`, `.tfvars` files, or environment variables (`TF_VAR_` prefix).
- **outputs** = return values. computed after apply, consumable by other modules or ci scripts.
- **locals** = private helper values for intermediate computations. callers never see these.

this distinction becomes important when composing modules: variables define the public interface, outputs define what the module provides, locals are implementation details.

### state

state is the most operationally important concept in terraform. the state file is a json document that maps every resource in your config to the corresponding real-world infrastructure object.

critical things to know:
- without state, terraform cannot know what exists. if you lose the state file, you have to manually import every resource one by one.
- for teams, a remote backend is non-negotiable. we use s3 for storage and dynamodb for locking.
- state locking prevents two engineers (or two ci jobs) from applying simultaneously and corrupting state.
- the state file contains sensitive data (connection strings, sometimes passwords). it should be encrypted at rest and access-controlled.
- backend configuration blocks in terraform cannot use variables or expressions — this is one of the language's most frustrating limitations, and it's a key reason terragrunt exists (more on this later).

### modules

a module is just a directory of `.tf` files. your root project is itself a module (the "root module"). you call other modules using a `module` block, passing variables as arguments.

- the public terraform registry has thousands of modules for common patterns (vpcs, ecs clusters, rds instances)
- we maintain an internal `terraform-modules` repository that encodes our company's opinions: tagging standards, security group defaults, naming conventions, monitoring integration
- our internal modules use a three-tier hierarchy:
  - **1-core**: foundational utilities (tagging, labeling)
  - **2-component**: reusable building blocks (alb, ecr)
  - **3-service**: complete solutions that compose core and component modules
- the tiering enforces dependency direction: services can depend on components, but components cannot depend on services

## 6. real hcl

this section transitions from concepts to practice. the slide shows a realistic ec2 instance example — not a hello-world, but the kind of code you'd actually encounter in a production codebase.

### variables and locals

walk through top to bottom:
- the `locals` block defines shared values: environment name, app name, and a `common_tags` map
- the `common_tags` pattern is ubiquitous in production terraform: define tags once, apply to every resource. this ensures consistent metadata for cost allocation, ownership tracking, and operations.
- `variable "instance_type"` has a sensible default (`t3.micro`); `variable "ami_id"` is required (no default)
- locals = internal implementation details. variables = public interface.

### resource and outputs

patterns to point out:
- `module.vpc.private_subnets[0]` references an output from a vpc module defined elsewhere. the ec2 resource doesn't need to know how the vpc was created — just that it exposes subnet ids. this is how modules compose.
- `merge(local.common_tags, { Name = "..." })` combines common tags with resource-specific tags
- `lifecycle { create_before_destroy = true }` tells terraform to create the replacement before destroying the old one during updates — standard pattern for minimizing downtime
- outputs export the instance id and private ip for consumption by other modules or ci scripts

## 7. terragrunt

this is the most important section for the audience. spend the most time here.

### what terragrunt is

terragrunt is a thin wrapper around terraform, built by gruntwork. it does not replace terraform — it runs terraform under the hood. its purpose is to solve the configuration management problems that emerge when you scale terraform across multiple environments and services.

meta-note for the talk: this very presentation is managed by terragrunt. the terraform module lives in `modules/presentation/` and a `terragrunt.hcl` wrapper in `live/presentation/` provides the inputs and inherits the root config. it's a small but working example of the patterns you're about to explain.

### what breaks at scale

the core problem is duplication. when you have multiple services (auth, api, events, etc.) each deployed across multiple environments (dev, prod), you end up with dozens of directories that each need nearly identical backend configs, provider blocks, and variable patterns.

at our scale — roughly 10+ services across 2 environments — that means 20+ copies of the same boilerplate. maintaining consistency is tedious and error-prone.

terraform also has no built-in mechanism for dependencies between separate state files (stacks). if your ecs service depends on a vpc, you have to manually ensure the vpc is applied first.

terragrunt addresses all of this through: configuration inheritance, code generation, and a dependency graph system.

### root config inheritance

one root `terragrunt.hcl` defines the remote state backend and provider config. every child module inherits this with a single `include` block.

how it works:
- the `remote_state` block defines the s3 backend once
- the key uses `path_relative_to_include()`, which automatically namespaces state files based on directory path
- `generate` blocks write `backend.tf` and `provider.tf` into the working directory before terraform runs
- terraform never sees terragrunt at all — it just sees normal `.tf` files

this is a key architectural point: terragrunt is a preprocessor, not a replacement. anything terraform can do, you can still do.

### environment layout

our directory structure encodes the environment model:

```
sre/
├── terragrunt.hcl              (root config)
├── dev/
│   ├── account.terragrunt.hcl  (aws account id + profile)
│   └── us-east-1/
│       ├── region.terragrunt.hcl
│       └── dev/
│           ├── environment.terragrunt.hcl
│           ├── vpc/terragrunt.hcl
│           ├── ecs/terragrunt.hcl
│           └── rds-backend-cluster/terragrunt.hcl
└── prod/
    ├── account.terragrunt.hcl  (different aws account)
    └── us-east-1/
        └── prod/
            └── (same components)
```

key insights:
- the hierarchy itself encodes context: account → region → environment → component
- each layer has its own `.hcl` file setting contextual values
- dev and prod live in entirely separate aws accounts (aws best practice for blast-radius reduction)
- promoting from dev to prod = same terraform module source, different inputs and aws account
- the leaf `terragrunt.hcl` files are intentionally minimal: just an include, a source, and inputs

### generated backends

this is one of terragrunt's biggest quality-of-life improvements.

the problem: terraform backend configuration blocks cannot use variables or expressions. this is because the backend needs to be resolved before terraform evaluates anything else. so every module needs a hardcoded backend block.

terragrunt's solution: generate the `backend.tf` file with correct values before terraform runs. the state key is derived from the directory path automatically, so `dev/vpc` and `prod/ecs` each get isolated state without manual config.

bonus: terragrunt will create the s3 bucket and dynamodb lock table if they don't exist, simplifying environment bootstrapping.

### cross-stack dependencies

`dependency` blocks express relationships between separate terraform stacks.

```hcl
dependency "vpc" {
  config_path = "../vpc"
  mock_outputs = {
    vpc_id             = "vpc-mock-12345"
    private_subnet_ids = ["subnet-mock-1", "subnet-mock-2"]
  }
  mock_outputs_allowed_terraform_commands = ["plan", "validate"]
}
```

how it works:
- when you run `terragrunt run-all apply`, terragrunt builds a dependency graph and applies modules in the correct order (vpc first, then ecs)
- without terragrunt, you'd use `terraform_remote_state` data sources and manually ensure apply order — fragile and doesn't scale
- `mock_outputs` let you run `terraform plan` on a module before its dependency has been applied — critical for ci pipelines
- mocks are restricted to `plan` and `validate` only; during `apply`, real outputs are required

in our codebase, dependency blocks are everywhere: api services depend on albs, albs depend on vpcs, database user modules depend on rds clusters.

## 8. ci/cd at tabs

this section shows how everything comes together in practice.

### the pr workflow

the rule: nobody runs `terragrunt apply` from their laptop. all infrastructure changes go through code review.

the flow:
1. engineer opens a pr that modifies terraform/terragrunt files
2. github actions runs `terragrunt plan` automatically
3. plan output is visible in the workflow run logs — reviewers check the actions tab
4. branch protection requires the plan workflow to pass before merge
5. on merge to main, `terragrunt apply-all` runs automatically
6. applies only trigger on the main branch

escape hatch: adding a `deploy-dev` label to a pr triggers a dev apply without merging. useful for testing infrastructure changes in a real environment before committing.

production applies are more strictly gated — they only run on merge to main, and only when files in the prod terragrunt directory have changed.

### matrix deploys

rather than deploying everything on every push, a `detect-changes` job analyzes the git diff to determine which services changed, then generates a matrix. the deploy job fans out across only those services.

so if only `auth-service` changed, only `auth-service` gets planned/applied. each service has its own terragrunt directory with components (alb, waf, api, dns), and `terragrunt run-all` handles dependency ordering between them.

we have three separate workflow files:
- `_dev.yaml`: dev deploys (build container + apply infra)
- `_prod.yaml`: prod deploys (apply infra only, uses the container image already built in dev)
- `_artifact.yaml`: ecr repository management

reusable steps come from a shared `cicd-templates` repository maintained by the platform team. aws authentication uses oidc — no static credentials stored in github secrets.

## likely questions and how to handle them

**"should we use opentofu instead?"**
honest answer: it's a valid option and good insurance, but the ecosystem is still overwhelmingly oriented around hashicorp terraform. the bsl doesn't affect our use case. switching would be low-effort if we ever needed to.

**"what about pulumi / cdk?"**
pulumi uses real programming languages instead of hcl. cdk generates cloudformation. both are valid approaches. terraform's advantages for us are: the massive provider ecosystem, the declarative model (harder to write bugs that create unintended side effects), and the fact that the entire team already knows it.

**"what happens if state gets corrupted?"**
s3 versioning is enabled on the state bucket. you can restore a previous version. for more serious issues, `terraform import` lets you re-associate resources with state entries. it's painful but recoverable.

**"why not just use cloudformation?"**
cloudformation only works with aws. terraform works with any provider. even if you're 100% aws today, terraform gives you the option to add non-aws resources (datadog monitors, github repos, pagerduty schedules) in the same workflow.

**"how do we handle secrets in terraform?"**
we don't store secrets in terraform state or config. secrets live in aws secrets manager or parameter store, and terraform references them via data sources. the state file can still contain sensitive attributes, which is why it's encrypted and access-controlled.
